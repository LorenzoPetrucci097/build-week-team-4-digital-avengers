Sezione 01

Executive Summary

Contesto

Il team Digital Avengers ha affrontato il task bonus del Giorno 3 della Build Week Epicode: una CTF (Capture The Flag) di tipo Black Box su un host virtuale (192.168.0.10). Lo scenario simula un penetration test realistico con accesso zero-knowledge, il cui obiettivo finale era ottenere privilegi root e catturare la flag conclusiva. Tutti i test sono stati condotti in ambiente di laboratorio autorizzato.

Obiettivo Principale

Compromettere completamente il sistema target attraverso una catena di vulnerabilità — recon, web exploitation (SQLi), analisi steganografica, Port Knocking, movimento laterale e privilege escalation — fino all'acquisizione della root flag.

Risultati Chiave

  • ✔ DONE Root flag acquisita: FLAG{misfatto_compiuto_999_root}
  • ✔ DONE SQL Injection su /oldsite/login.php — dump completo credenziali DB (4 utenti, hash BCrypt)
  • ✔ DONE Password cracking con Hashcat: milena:darkprincess
  • ✔ DONE Accesso SSH via brute-force Hydra: user:harry (porta 2222)
  • ✔ DONE Decodifica Brainfuck e ricostruzione sequenza Port Knocking (6 porte)
  • ✔ DONE Analisi steganografica theta-logo.jpg → poesia.txt (passphrase: accio)
  • ✔ DONE Estrazione chiave RSA da theta-key.jpg → accesso root senza password

Rischio Residuo: Il sistema presenta una catena di criticità CRITICHE: SQL Injection non mitigata, credenziali deboli, hash BCrypt con wordlist comune, dati sensibili in steganografia e Port Knocking come unica barriera pre-SSH. Priorità immediata: patch SQLi con prepared statements, rimozione dei dati sensibili dagli asset web, rotazione delle chiavi SSH e implementazione di autenticazione MFA.

Sezione 04

Threat Model & Risk Assessment

4.1 Asset & Attack Surface

L'host 192.168.0.10 espone: server web Apache 2.4.52 (porta 80), SSH OpenSSH 8.9p1 (porta 2222, poi 22 post-knock), FTP Synology (porta 21), PPTP VPN (1723), e servizi wrappati (8080, 8443). La superficie primaria è il layer web con due applicazioni PHP (sito moderno e /oldsite), entrambe con form di autenticazione.

4.2 Threat Actors & TTPs Ipotizzate

Scenario simulato: attaccante esterno con accesso alla rete locale. Profilo: penetration tester intermediate-advanced. TTPs principali: T1595 (Active Scanning), T1190 (Exploit Public-Facing App), T1110.001 (Brute Force), T1078 (Valid Accounts), T1548 (Privilege Escalation).

Sezione 05

Processo Operativo Completo

Approccio Metodologico

Metodologia applicata: PTES (Penetration Testing Execution Standard) con elementi di Cyber Kill Chain e OWASP Testing Guide v4.2 per il layer web. Framework di mapping: MITRE ATT&CK v14. Di seguito il processo operativo completo, fase per fase, con comandi eseguiti, risultati ottenuti ed evidenze.

FASE 1 — Ricognizione & Enumerazione

1.1 Identificazione Host (netdiscover)

L'attività ha avuto inizio con l'identificazione degli host attivi all'interno del segmento di rete locale tramite richieste ARP.

$ sudo netdiscover -r 192.168.0.0/24

Risultato: individuato l'host target all'indirizzo IP 192.168.0.10, identificato come macchina virtuale Oracle VirtualBox (MAC: 08:00:27:72:5E:0F).

Evidence · Fase 1.1 Output netdiscover — Host Discovery

1.2 Analisi Nmap (SYN Stealth Scan)

Scansione approfondita con rilevamento OS (-O) e versioni servizi (-sV).

$ nmap -sS -sV -O -T3 192.168.0.10

Porte aperte rilevate:

PortaStatoServizioVersione / Note

Anomalia Porta 21: Nmap segnala pacchetti RST — indica possibile firewall, IPS o configurazione honeypot.

SSH su porta 2222: Misura di Security through Obscurity. OS: Linux Kernel 4.x/5.x (Ubuntu).

FASE 2 — Web Enumeration & Information Gathering

2.1 Directory Brute-Forcing (ffuf)

Con la porta 80 aperta, l'analisi si è spostata sull'identificazione di file e directory nascoste.

  -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt \

FASE 3 — Vulnerability Assessment: SQL Injection

3.1 Identificazione della Vulnerabilità

I test di input sui form di login hanno rivelato una gestione non sicura delle query al database. L'inserimento del payload classico ' OR 'a'='a ha generato un errore fatale nel backend del sito old, confermando la possibilità di manipolare la query SQL originale. Il sistema è stato identificato come basato su MariaDB.

Evidence · Fase 3.1 Errore SQL rivelatore su /oldsite/login.php

3.2 Sfruttamento (Union-Based SQLi)

È stata utilizzata una tecnica di Union-Based SQL Injection per esfiltrare dati dal database attraverso il form di login. La tecnica di inversione delle colonne è risultata efficace per allineare i tipi di dati.

' UNION SELECT database(),2#

3.3 Esfiltrazione dei Dati (Data Dumping)

Attraverso una serie di query mirate, è stata ricostruita l'intera struttura del database:

' UNION SELECT group_concat(table_name),2
  FROM information_schema.tables
  WHERE table_schema=database()#
' UNION SELECT group_concat(column_name),2
  FROM information_schema.columns
  WHERE table_name='users'#
' UNION SELECT group_concat(username,':',password),2
  FROM users#

FASE 6 — Esfiltrazione & Steganografia Avanzata

6.1 Esfiltrazione theta-key.jpg

Il file .theta-key.jpg.bk nella home di Luca aveva permessi restrittivi. Il blocco è stato aggirato sfruttando i permessi del gruppo shared, spostando una copia nella directory condivisa.

6.2 Estrazione della Chiave Privata RSA

Sulla macchina Kali, l'immagine è stata sottoposta ad analisi steganografica utilizzando la passphrase recuperata dal file di swap.

$ steghide extract -sf theta-key.jpg -p "ai(q4P7>(Fw9S3P"

Risultato critico: estrazione del file id_rsa — una chiave privata SSH per l'utente root.

FASE 7 — Privilege Escalation: Root

7.1 Hardening della Chiave e Accesso Root

La chiave RSA estratta permette di bypassare completamente l'autenticazione basata su password.

$ chmod 600 id_rsa
$ ssh -i id_rsa root@192.168.0.10

Sezione 09

Lessons Learned & Mitigazioni

9.1 Cosa ha funzionato

  • Approccio metodico PTES: recon prima dell'exploitation, nessun passaggio saltato
  • Decomposizione parallela del task (web + steg + recon in contemporanea)
  • ffuf con estensioni multiple per scoperta rapida di /oldsite e /tmp
  • Intuizione sulla passphrase accio dal contesto narrativo Harry Potter
  • Coordinamento team: ogni membro ha guidato almeno una fase specifica

9.2 Criticità & Miglioramenti

  • Tempo speso sul Cookie /oldsite (base64 parziale) senza risultati utili — skip più rapido
  • Alcune analisi PNG con stegseek non conclusive — integrare zsteg come tool alternativo
  • La porta 1700 (da Hydra) nel Port Knocking non era intuitiva — documentare meglio la catena
  • Permessi id_rsa sbagliati al primo tentativo — ricordare chmod 600 subito
